1 技術選型
🌐 HTML5 — 單一入口 index.html
🎨 純 CSS — 無框架，手寫 Design System
⚡ Vanilla JS — 無打包工具，無 npm
📊 Google Sheets API v4 — 唯讀資料庫
🚀 GitHub Pages — 零成本靜態托管

沒有 Node.js、沒有 bundler、沒有後端。開啟 index.html 本機就能跑；push 到 GitHub 即自動上線。

4 啟動流程
browser 載入 index.html
依序執行 data.js → sheets.js → app.js（<script> 順序）
data.js 先建立含 mock 資料的 APP_DATA 與 MATURITY_LEVELS
app.js 最後一行呼叫 init()
#content 顯示「載入資料中…」loading 畫面
loadFromSheets() 並行讀取 6 個分頁
使用 Promise.allSettled：任一分頁失敗不影響其他分頁；
成功的分頁覆寫 APP_DATA 對應欄位，失敗的保留 mock 資料
更新右上角日期標籤
Sheets 成功 → 顯示今日日期；失敗 → 顯示「本地資料」
navigate('dashboard') 渲染首頁
sidebar 高亮第一個 nav-item，renderDashboard() 將 HTML 注入 #content
5 SPA 路由設計

沒有 URL hash / History API。Sidebar 每個 <a> 直接呼叫 navigate('頁面名稱')， 函式清空 #content 後呼叫對應 render 函式，整頁重新 innerHTML。

7 關鍵設計決策
為什麼不用 React / Vue？
靜態 HTML + innerHTML 渲染
單人維護、無 CI/CD 建構需求、GitHub Pages 直接托管靜態檔。 框架帶來的 bundler / node_modules 複雜度超過本專案需要。 缺點：篩選後切換頁面會重置狀態。
為什麼不用 Firebase / Supabase？
Google Sheets API（唯讀）
只有一位管理員寫入資料，主管唯讀。 Sheets 本身就是最熟悉的編輯介面，免去後台管理頁開發。 敏感性低，無需即時多人同步。
為什麼 data.js 的 mock 資料要保留？
Fallback 機制
Sheets API 需要網路、API key 限制、分頁若格式錯誤會失敗。 Promise.allSettled 保證某分頁失敗不影響其他分頁， 且 mock 資料確保離線 / key 過期時畫面不空白。
API Key 安全性？
HTTP Referrer 限制
Key 限制只允許來自 https://markcqhsu.github.io/* 的請求。 Sheets 本身唯讀（無寫入權限），資料非高度敏感， 此方案在安全性與便利性間取得平衡。
成熟度等級顏色怎麼決定？
MATURITY_LEVELS 常數集中管理
L0–L5 的 label / desc / color 定義在 data.js 的 MATURITY_LEVELS 物件。 所有頁面透過 lv(level).color 取色，改一處全站同步， 不在 CSS 寫死顏色。
地區分組怎麼運作？
regions.group 動態分組
集團總覽的地區卡用 regions.group 欄位動態 groupBy， 不在 JS 寫死「台灣 / 中國 / 東南亞」。 新增地區只需在 Sheets 填 group，無需改程式。
8 已知限制
限制原因如果未來需要解決
篩選狀態在切換頁面後重置 每次 navigate() 整頁重渲染 加 URL hash 記錄狀態，或改用前端路由框架
Sheets 資料無快取，每次開頁面重新抓取 無 Service Worker / localStorage 加 localStorage cache + TTL（如 5 分鐘）
無法即時推播更新（需手動重新整理） 靜態前端，無 WebSocket 加定時 reload 或 Google Sheets 的 Pub/Sub webhook
API Key 明碼暴露在原始碼 靜態部署無法隱藏 改用 OAuth 或後端 proxy；現況靠 Referrer 限制降低風險
資料量大時 DOM 渲染會慢 innerHTML 全量重繪，無虛擬 DOM 加虛擬捲動（Virtual Scroll）或分頁
